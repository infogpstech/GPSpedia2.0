NUEVOS HALLAZGOS DE AUDITORÍA (services/catalog/catalog.js)
-------------------------------------------------

### Auditoría Realizada: 05/01/2026

- **Propósito:** Análisis final del `catalog-service` antes de su formalización para producción.
- **Estado General:** El servicio es robusto, cumple con la arquitectura definida y está funcionalmente completo. Los siguientes puntos se registran para futuras mejoras y para tener un conocimiento claro de los riesgos operativos.

#### **Riesgos Identificados**

1.  **Dependencia Estricta de Mapas de Columnas (Hardcoding):**
    *   **Observación:** El servicio depende de constantes (`COLS_CORTES`, `COLS_LOGOS_MARCA`, etc.) que mapean nombres de campos a posiciones de columna fijas.
    *   **Riesgo:** Si un administrador modifica la estructura de la hoja de cálculo (ej. inserta, elimina o mueve una columna) sin actualizar simultáneamente y de forma precisa estos mapas en el código, el servicio fallará de manera silenciosa. Servirá datos incorrectos (ej. asignando un `modelo` a el campo `marca`) sin generar un error explícito. Este es el principal riesgo operativo del servicio.

2.  **Rendimiento de Sugerencias a Gran Escala:**
    *   **Observación:** La función `handleGetSuggestion` utiliza el algoritmo de `levenshteinDistance` contra una lista de todos los valores únicos de una columna.
    *   **Riesgo:** A medida que la base de datos crezca a miles de entradas, el cálculo de la distancia para cada sugerencia podría volverse lento y exceder los límites de tiempo de ejecución de Google Apps Script, haciendo que la función falle.

#### **Deuda Técnica Registrada**

1.  **Función `handleGetCatalogData` Monolítica:**
    *   **Observación:** Esta función es responsable de obtener, procesar y mapear los datos de cuatro hojas de cálculo diferentes (`Cortes`, `LogosMarca`, `Tutorial`, `Relay`).
    *   **Deuda:** Su gran tamaño y múltiples responsabilidades la hacen difícil de leer y mantener.
    *   **Recomendación (Tarea Futura):** Refactorizar esta función dividiéndola en sub-funciones más pequeñas y enfocadas (ej. `fetchCortesData`, `fetchLogosData`, etc.) para mejorar la claridad.

2.  **Lógica Compleja de Ordenamiento de Cortes:**
    *   **Observación:** La lógica para reordenar los cortes de un vehículo según su popularidad (`utilCorteX`) dentro de `handleGetCatalogData` es funcional pero compleja de seguir. Implica crear un objeto temporal, ordenar un array de índices y luego reconstruir el objeto del vehículo.
    *   **Deuda:** Dificulta el mantenimiento futuro de esta funcionalidad.
    *   **Recomendación (Tarea Futura):** Añadir comentarios detallados a este bloque de código o refactorizarlo a una función auxiliar (`sortVehicleCuts`) para aislar su complejidad.

#### **Observaciones Adicionales**

1.  **Cumplimiento Arquitectónico (Positivo):**
    *   **Observación:** El servicio cumple de manera excelente con la regla arquitectónica de manejar solo `fileId`s para las imágenes, gracias a la robusta función `normalizeAndValidateImageId`. Esto es un pilar de la estabilidad del sistema.

2.  **Uso de Caché (Positivo):**
    *   **Observación:** La implementación de `CacheService` es correcta y vital para el rendimiento de la aplicación, evitando lecturas innecesarias a la base de datos.

3.  **Ineficiencia en `handleGetDropdownData`:**
    *   **Observación:** Esta función lee todos los datos de la hoja `Cortes` varias veces (una por cada campo de dropdown) para obtener listas de valores únicos.
    *   **Recomendación (Tarea Futura):** Optimizar esta función para que lea el `dataRange` una sola vez en memoria y luego genere todos los conjuntos de valores únicos a partir de ese array.

  - **Estado:** EN PROGRESO
  - **Urgencia:** No Negociable
  - **Tipo:** Requerimiento de Arquitectura
  - **Descripción:** Diseñar e implementar una solución para manejar las imágenes de Google Drive a través de un proxy en el backend.

  - **Restricción:** El frontend NO debe recibir URLs directas de Google Drive. El tamaño de las imágenes debe ser controlado exclusivamente por CSS, sin usar parámetros de URL (`=w...`, `sz`, etc.).

- **Tarea 28: Endurecer `catalog-service` para Manejo de Imágenes**
  - **Urgencia:** Crítica
  - **Tipo:** Violación de Arquitectura / Bug
  - **Descripción:** Durante la auditoría final, se detectó que `services/catalog/catalog.js` todavía contiene lógica para transformar URLs de Google Drive (`convertirAGoogleThumbnail`). Esto viola la arquitectura final, que estipula que solo el `image-service` debe manejar interacciones con Drive y que el `catalog-service` solo debe pasar `fileId`s crudos. Esta transformación incorrecta es la causa raíz de los fallos de carga de imágenes.
  - **Requerimiento Técnico:**
    1.  **Eliminar `convertirAGoogleThumbnail` de `catalog.js`:** El servicio debe dejar de transformar los IDs.
    2.  **Implementar Normalización en `catalog.js`:** Añadir una nueva función `normalizeAndValidateImageId` que reciba el valor de la celda de imagen. Esta función debe ser capaz de manejar tanto un `fileId` limpio como una URL completa de Drive, extrayendo el ID en el segundo caso. Si el dato es inválido, debe devolver `null`.
    3.  **Aplicar Normalización:** La función `mapRowToObject` debe usar `normalizeAndValidateImageId` para limpiar todos los campos de imagen antes de enviar los datos al frontend.
  - **Impacto Potencial:** Resuelve definitivamente el bug de carga de imágenes, alinea el servicio con la arquitectura final y lo hace más robusto ante datos inconsistentes en la hoja de cálculo.
